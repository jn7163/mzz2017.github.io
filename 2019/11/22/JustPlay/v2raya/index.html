V2RayA

V2RayA是一个支持全局透明代理的V2Ray Linux客户端。

在今年国庆前后，机场和梯子出现了大面积塌方，当时还在用ss的我被迫认识了v2ray，可问题也随之而来，一直用Linux的我，竟然没找到一个在Linux上体验很棒的客户端。后来发现了v2rayL，界面上实际上已经不错了，也满足我的一般需求，但v2rayL的作者在考研，更新速度没有那么快，当时还有少量bug存在以及有诸多功能没有实现，于是想法自己实现一个，接下来就有了V2RayA。

项目地址：https://github.com/mzz2017/V2RayA

前端 demo: https://v2raya.mzz.pub

功能清单

目前已完成基本功能，项目正在逐步开发中。

已支持：

  • 全局透明代理（不支持 docker 模式）
  • V2Ray 服务控制
  • 导入 vmess、ss、订阅地址
  • websocket、kcp、tcp、http、tls、shadowsocks 全支持
  • 测试节点 Ping 时延
  • 二维码、地址分享
  • 同时开放 socks5、http、带 PAC 的 http 三个端口（PAC 模式可选 GFWList、大陆白名单）
  • 服务端启动自动检查 PAC、订阅更新
  • 多路复用、TCPFastOpen 支持
  • 以service方式启动

待开发：

  • 检查版本更新
  • 手动添加/修改节点
  • 测试节点 HTTP 时延
  • 自定义 PAC 路由规则
  • QUIC 协议支持
  • 服务端端口号配置、前端可指定服务端地址
  • 登陆与安全

使用

如下使用方法：

  1. 使用apt-get安装（debian、ubuntu）

    请确保已正确安装 v2ray-core

    官方提供了 Linux 下的一键安装脚本：

    运行下面的指令下载并安装 V2Ray。当 yum 或 apt-get 可用的情况下，此脚本会自动安装 unzip 和 daemon。这两个组件是安装 V2Ray 的必要组件。如果你使用的系统不支持 yum 或 apt-get，请自行安装 unzip 和 daemon

    准备完毕后：

    V2RayA服务端正常运行后，就可在Web-GUI使用了（或手动部署 Web-GUI）。

  2. 使用pacman安装（archlinux、manjaro）

    由于v2raya发布在AUR中，而pacman不支持AUR，因此建议使用主流的yay或yaourt作为替代方案

    当yay或yaourt可用时，可通过yay或yaourt安装v2raya

    V2RayA服务端正常运行后，就可在Web-GUI使用了（或手动部署 Web-GUI）。

  3. 使用Release二进制文件/安装包（支持常见linux系统）

    请确保已正确安装 v2ray-core

    官方提供了 Linux 下的一键安装脚本：

    运行下面的指令下载并安装 V2Ray。当 yum 或 apt-get 可用的情况下，此脚本会自动安装 unzip 和 daemon。这两个组件是安装 V2Ray 的必要组件。如果你使用的系统不支持 yum 或 apt-get，请自行安装 unzip 和 daemon

    准备完毕后，可下载Releases中的二进制文件启动V2RayA服务端，或下载安装包进行安装。

    V2RayA服务端正常运行后，就可在Web-GUI使用了（或手动部署 Web-GUI）。

  4. 当然，你也可以选择拉取源码，通过 golang 启动：

    该方法同样需要正确安装v2ray-core，详情见上

    注意，尽管 golang 具有交叉编译的特性，但由于项目使用了 linux commands，导致该方法不支持 windows。若想在 windows 体验，可借助 Docker 或 WSL。

  5. 拉取源码，使用 docker-compose 部署。该方法不支持全局透明代理。

  6. 使用 docker 命令部署。该方法不支持全局透明代理。

    我们同步发行Docker 镜像，如果无法使用 docker-compose，可以参考docker-compose.yml并使用 docker 命令自行搭建。

默认使用的四个端口分别为：

2017: V2RayA 后端端口

20170: SOCKS 协议

20171: HTTP 协议

20172: 带 PAC 的 HTTP 协议

在不同运行环境下程序表现将不同

由于 docker 容器对 systemd 的限制性，在 docker 中将采用 pid 共享进程命名空间，volumes 共享存储空间，更新配置后通过结束进程触发 v2ray 容器的重启来更新配置，以无 inbounds 的配置代替断开连接，这是一种折中方案，会有如下影响：

  1. 在更换配置时略有卡顿
  2. 无法使用全局透明代理

在宿主环境下以 sudo 权限运行将不受此限制。

支持 Windows、MacOS 吗

目前仅在 Linux 进行过测试，并计划优先适配 Linux。目前尚未验证在 Windows 及 MacOS 上存在的问题。

实际上 Windows 和 MacOS 上已经存在很多优秀的 V2Ray 客户端，若无特殊需求，建议选择这些客户端。

其他话

在项目开发的这段时间中，我陆续还发现了一些优秀的V2Ray Linux客户端，在最开始，我最多认为自己的项目顶多是界面更好看一点，而在功能上和他们并无优势（实名感谢美院Jenny同学帮忙调色）。

让我改变想法的是我尝试将V2Ray的路由功能进行实现，将PAC模式下大陆白名单和GFWList模式开发出来后，我觉得V2RayA开始有点东西了。

真正让我觉得V2RayA稍微没那么黯淡时候，是我将全局透明代理实现了之后。实际上，全局代理是很多人的需求，也是我的需求，很多软件不支持设置socks代理端口，这时就很期望ss或者v2ray不仅是开放一个socks端口，而是可以用于全局代理。

V2RayA本来是计划完全基于docker的，所以你会看到在代码中单独针对docker环境进行了适配，甚至单独提出了docker方案。而在考虑实现全局透明代理时，我发现docker模式下不可能支持这一模式，于是对重心整改，将本来依赖于redis数据库的部分也改成依赖于文件系统，简化非docker模式下的安装配置。从docker脱离出来，才有后面V2RayA易用性的进一步提升；而docker模式的尝试，也使得V2RayA的适用性更强（#2）。

更多方面，我希望V2RayA能更方便得到下载和使用，所以单独肝了两三天的travis，希望它能够通过apt、yum、pacman等包管理器直接下载。而出于money的考虑，我又不希望在自己服务器上建软件源，万一以后火了，流量烧得哗哗的我不得心疼吗。最后虽然我travis配置写得很烂，但至少实现了上述的想法。

上面哪一个点单独拿出来都没办法让V2RayA在Linux客户端里独树一帜，但加起来之后，终于就有那么点味道了。无论怎样，V2RayA始终是一个基于V2Ray的客户端，是一个“小玩意儿”，是在我强迫症的毛病下惯出来的一个小玩意儿。

好了，我已经连续通宵半个月了，下周该去深圳了，我要好好把时差倒过来了。

晚安。
